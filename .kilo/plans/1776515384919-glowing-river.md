# Plan: Full Redesign of Admin User Management Page (`admin/users.php`)

## Goal
Redesign the entire admin user management experience, with primary focus on the table UI, to make user edits faster, safer, and easier while preserving all existing backend behavior and endpoint compatibility for:

- `http://localhost/library_system_online/admin/users.php`

## Non-Negotiables
- Keep all existing server actions and CSRF protection working.
- Preserve role/security constraints (superadmin protection, last-active-admin guard, role-based verification policy).
- No DB schema changes.
- Preserve query/filter/pagination behavior.
- Maintain compatibility with existing SweetAlert flows and flash messaging.

## Current Pain Points (Observed)
- Row-level action area is visually heavy and causes scanning fatigue.
- Editing controls are duplicated per row, increasing clutter and vertical expansion.
- Table hierarchy is weak: identity, role, verification, status, and actions compete visually.
- Bulk action feedback is present but not strongly integrated into row state clarity.
- Styles are spread across overlapping CSS sections, increasing cascade conflicts.
- Mobile/tablet behavior is functional but still difficult for high-frequency admin operations.

## Redesign Direction (Default)
Use a **clean, operations-first admin console** layout aligned to existing brand tokens:

1. Strong data table hierarchy for quick scanning.
2. Single reusable edit surface (drawer/panel) instead of dense per-row forms.
3. Clear quick actions in-row (activate/deactivate + open manage panel).
4. Strong visual state cues for verification/eligibility/protected accounts.
5. Consistent desktop and mobile workflows.

## Implementation Plan

### Phase 1 — Information Architecture and Markup Refactor (`admin/users.php`)
1. Keep server POST handlers intact (including recent verification and role-policy logic).
2. Rebuild table markup into a v2 structure with stronger semantic grouping:
   - Identity block (name, email, optional metadata)
   - Access block (role badge + role hint)
   - Trust block (verification + account status)
   - Operations block (quick action + manage button)
3. Replace per-row expanded forms with:
   - A compact row action trigger (`Manage`)
   - A single shared “Manage User” drawer/panel outside the table
4. Keep hidden-safe fallbacks where appropriate so no action becomes JS-only.
5. Preserve bulk-select checkboxes and hidden selected user submission behavior.

### Phase 2 — New Edit Workflow Surface (Drawer/Panel)
1. Add one reusable manage panel containing sections:
   - Credentials
   - Access policy (verification + role)
   - Password reset
   - Account status controls
   - Permanent delete
2. Populate panel from selected row dataset attributes (`data-*`), not duplicated full forms per row.
3. Submit existing server actions with original field names:
   - `update_user_credentials`
   - `update_user_verification`
   - `role_change`
   - `admin_reset_user_password`
   - `activate` / `deactivate`
   - `delete_user`
4. Keep superadmin rows non-editable and clearly marked as protected.
5. Preserve focus return and post-redirect row highlight behavior.

### Phase 3 — Table UI Redesign (Primary Focus)
1. Rebuild visual table system with stronger tokens and consistency:
   - Sticky header with clearer contrast
   - Balanced column widths
   - Reduced row noise
   - Distinct state chips for role/verification/status
2. Add row states:
   - Selected
   - Focused-after-update
   - Ineligible for role promotion (needs verification)
   - Protected
3. Improve operations UX:
   - Fixed-width action zone
   - Primary quick action button (activate/deactivate)
   - Secondary manage button
4. Improve bulk-action integration:
   - Eligibility count surfaced in toolbar status
   - Optional subtle row tint for skipped users under role bulk actions

### Phase 4 — CSS Consolidation and Cleanup (`assets/css/libris.css`)
1. Create a clearly scoped admin-users-v2 section.
2. Remove/neutralize conflicting duplicate selector blocks for users page styles.
3. Normalize spacing, typography scale, badge rhythm, and panel borders.
4. Ensure robust responsive breakpoints:
   - Desktop: high-density table view
   - Tablet: adjusted spacing and controls
   - Mobile: card-like row stacks with preserved actions
5. Honor reduced-motion preferences for animations/highlights.

### Phase 5 — JS Interaction Refinement (`admin/users.php` inline script or extracted file)
1. Introduce a single source of truth for selected/current user state.
2. Add drawer open/close, focus trap, and keyboard handling (`Esc`, tab loop).
3. Keep existing client-side guardrails:
   - No-change detection
   - Password mismatch checks
   - Role/verification policy checks before submit
4. Keep SweetAlert confirmations, and ensure submit button reset on cancel/errors.
5. Keep bulk action hidden-input generation and role eligibility filtering.

### Phase 6 — Accessibility and Usability Hardening
1. Ensure all controls have labels and clear `aria-describedby` hints.
2. Maintain live region announcements for selection and operation readiness.
3. Improve keyboard navigation order from table row to manage drawer.
4. Ensure color contrast for badges and status chips remains compliant.

### Phase 7 — Verification and Regression Testing
1. Syntax check:
   - `php -l admin/users.php`
2. Functional checks:
   - Search/filter/pagination
   - Bulk actions (including role eligibility skips)
   - Credentials/verification/role/password/status/delete operations
   - Superadmin protection and last-active-admin safeguards
3. UX checks:
   - Desktop/tablet/mobile table readability
   - Manage panel workflow speed
   - Post-action focus and row highlight
4. Endpoint compatibility check:
   - Confirm `admin/users.php` loads and actions redirect correctly.

## Files Planned for Update

## Acceptance Criteria
1. Page is visibly redesigned end-to-end, especially table UI and row action flow.
2. Admin can edit users with fewer steps and less visual clutter.
3. All existing security and role-based validation rules still enforce correctly.
4. Bulk actions remain accurate with explicit eligibility feedback.
5. Page remains fully compatible with `admin/users.php` backend behavior.

## Execution Order (Implementation)
1. Markup refactor and shared manage panel scaffold.
2. JS data binding and action submission behavior.
3. Table and panel CSS redesign + responsive tuning.
4. Regression pass for all user management actions.
