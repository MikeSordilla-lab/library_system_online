# Phase 1 Plan: Library Ticket/Receipt Feature (No POS Middleware)

## Scope and Constraints
- Implement **Phase 1 only**: web-based ticket/receipt generation, preview, and printing via browser/PDF.
- No POS middleware, no ESC/POS direct commands, no printer daemon/agent integration.
- Deliver a clear and stable base that can support future hardware integrations.

## 1) Receipt/Ticket Types and Required Content

### In-scope for Phase 1
1. **Checkout Receipt**
2. **Fine/Payment Receipt**
3. **Reservation/Event Ticket**
4. **Temporary Visitor Pass**

### Required common fields (all types)
- Library name + branch
- Receipt/Ticket number (human-readable unique ID)
- Internal UUID
- Issue date/time (timezone-aware)
- Channel (`staff` or `kiosk`)
- Issued by (staff ID/name) when applicable
- Patron identifier (masked)
- QR payload (for lookup/verification/check-in)
- Footer (policy/legal text)

### Type-specific fields
- **Checkout**: borrowed items, copy barcode, due dates, total item count.
- **Payment**: payment lines (fine reason), totals, payment method, transaction reference, remaining balance.
- **Reservation/Event**: event/room, schedule window, seat/slot if applicable.
- **Visitor Pass**: validity start/end, visitor display name (masked/optional), host/staff reference.

### Format variants in Phase 1
- Thermal-like web layout (58/80mm style via print CSS)
- A4 print layout
- PDF output for archival/download

## 2) Technical Design

## Architecture
- Backend REST endpoints for issue/fetch/render/reprint.
- Relational DB persistence for receipts + print job logs.
- Server-rendered HTML templates with print CSS.
- PDF rendering pipeline from same template data.
- Frontend print preview UI (browser print + PDF download).

### API Endpoints (Phase 1)
1. `POST /api/receipts`
   - Creates receipt record with immutable payload snapshot.
   - Supports idempotency via `Idempotency-Key`.
2. `GET /api/receipts/{receiptId}`
   - Returns details and render links.
3. `GET /api/receipts/{receiptId}/print`
   - Returns printable HTML.
4. `GET /api/receipts/{receiptId}.pdf`
   - Returns generated PDF (on-demand or cached).
5. `POST /api/receipts/{receiptId}/reprint`
   - Logs reprint event and returns same render links.
6. `GET /api/receipts/{receiptId}/qr`
   - Returns QR image or payload descriptor.

### Data model changes

#### `receipts`
- `id` (UUID, PK)
- `receipt_no` (unique, human-readable)
- `type` (`checkout|payment|event|visitor`)
- `branch_id`
- `patron_id` (nullable)
- `issued_by_user_id` (nullable)
- `channel` (`staff|kiosk|api`)
- `status` (`issued|voided|reprinted`)
- `format` (`thermal|a4|ticket`)
- `locale`, `timezone`, `currency`
- `payload_json` (immutable snapshot)
- `payload_hash` (HMAC or hash)
- `issued_at`, `created_at`, `updated_at`

#### `receipt_items`
- `id`, `receipt_id` (FK)
- `item_type`, `ref_id`, `title`
- `meta_json`
- `amount` (nullable)

#### `receipt_print_jobs`
- `id`, `receipt_id` (FK)
- `attempt_no`
- `target` (`browser|pdf|email`) for Phase 1 mostly `browser|pdf`
- `status` (`queued|success|failed`)
- `error_code`, `error_message`
- `requested_by_user_id`
- `requested_at`, `completed_at`

### Template and layout strategy
- Base template + partials by receipt type.
- Print CSS profiles by paper size/style (`thermal`, `a4`).
- Keep one canonical payload contract consumed by all templates.
- Include QR image and readable fallback text payload.

### Rendering strategy for Phase 1
- **Primary**: browser print from printable HTML route.
- **Secondary**: server-generated PDF for download/archive.
- No direct printer command streams in this phase.

### QR/Barcode approach
- Use QR first (lowest friction in web/PDF).
- Payload example: `LIB|<receipt_no>|<type>|<issued_at>|<sig>`
- Signature = HMAC over canonical fields to prevent tampering.

### Localization and format differences
- i18n labels via translation keys.
- Locale-aware date/currency formatters.
- Branch config for default paper profile and footer text.

## 3) Printer Integration for Phase 1
- **Enabled**: browser-native print dialog and PDF download.
- **Out of scope**: USB/serial/network spooler middleware, ESC/POS command output.
- **Fallbacks in scope**: email/PDF link optional (if existing mail service already present).

### Security considerations
- RBAC for issue/reprint/void actions.
- Signed or authenticated access to print/PDF endpoints.
- Short-lived URLs when exposing PDF links externally.

## 4) Workflow and UX

### Staff/admin flows
- Add `Issue Receipt` action at completion of checkout/payment/reservation/pass creation.
- Show post-success modal/page with:
  - Print preview
  - `Print`
  - `Download PDF`
  - optional `Email` (if available)

### Kiosk flow
- Auto-show receipt screen after transaction.
- Provide `Print`, `Download`, `Done` with auto-timeout.

### Reprint and batch
- Reprint from transaction detail page with reason capture.
- Batch printing (events/reservations) can be deferred if timeline is tight; keep API-compatible.

### Error handling
- If print dialog canceled: keep transaction successful, offer retry.
- If PDF generation fails: show clear message and retry action.
- Log all failures in `receipt_print_jobs`.

## 5) Data Integrity, Logging, and Auditing
- Generate unique `receipt_no` with branch/date sequence.
- Enforce idempotency on `POST /api/receipts`.
- Persist immutable payload snapshot for audit/legal consistency.
- Audit events:
  - `receipt_issued`
  - `receipt_viewed`
  - `receipt_printed`
  - `receipt_reprinted`
  - `receipt_voided`
- Reprint counters and actor attribution for abuse prevention.

## 6) Implementation Roadmap (Phase 1)

## Milestone A: Requirements + Schema (Small)
**Effort:** 2-3 days
- Finalize fields and legal footer content.
- Add DB migrations and indexes.
- Define numbering/idempotency strategy.

## Milestone B: Core API + Service Layer (Medium)
**Effort:** 3-5 days
- Implement create/fetch/reprint endpoints.
- Implement payload snapshot builder and hash/signature.
- Add audit event emission.

## Milestone C: Templates + Rendering (Medium)
**Effort:** 3-5 days
- Build base template and 4 type partials.
- Implement print CSS for thermal + A4.
- Add PDF endpoint and caching policy.

## Milestone D: Frontend Integration (Medium)
**Effort:** 3-6 days
- Integrate print actions in staff workflows.
- Add kiosk receipt screen.
- Add reprint entry points and reason UI.

## Milestone E: Hardening + UAT (Small/Medium)
**Effort:** 2-4 days
- Permission checks, localization validation, error UX.
- End-to-end smoke tests in staging.

### Testing strategy
- **Unit**: numbering, idempotency, payload signing, template helpers.
- **Integration**: issue -> fetch -> HTML print -> PDF render -> reprint.
- **UI/E2E**: staff flow and kiosk flow, error scenarios.
- **Cross-browser**: Chrome/Edge print layout checks.

### Deployment and rollback
- Release behind feature flag (e.g., `receipts_phase1_enabled`).
- Apply migrations first, enable by branch incrementally.
- Rollback by disabling feature flag while preserving data.

## 8) Security, Privacy, and Compliance Notes
- Print only minimum necessary personal data (mask patron identifiers).
- Exclude sensitive fields (full address, full phone, full card data).
- Apply retention policy for receipt payload and print logs.
- Support right-to-access/deletion workflows where legally required.
- Keep audit trails immutable and access-controlled.

## 9) Optional Enhancements (Post-Phase 1)
- Digital receipts via email/SMS with signed links.
- QR/NFC-assisted check-in.
- Receipt analytics dashboard (volume, reprint rate, failures).
- Expanded multilingual and branch-specific templates.

## Acceptance Criteria for Phase 1 Completion
- Staff can issue and print/download receipts for all 4 types.
- Kiosk can show and print/download ticket details.
- Reprint action exists with audit trail.
- All receipt records are uniquely identifiable and idempotent on create.
- Print/PDF output remains readable and consistent in supported browsers.
- No POS middleware or direct printer command integration introduced.
