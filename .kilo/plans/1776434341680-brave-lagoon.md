# Plan: Admin User Management CRUD Enhancement

## Goal
Enable `admin` users to manage account credentials for all user types **except superadmin**, specifically:
- Reset/change another user's password
- Update another user's credential fields (at minimum: `email`; optionally `full_name` depending on final requirement)

## Current State (from code inspection)
- `admin/users.php` currently supports create, role change, activate/deactivate, bulk update, delete, and superadmin protections.
- It does **not** currently support editing existing user credentials (`email` / `full_name`) or resetting another user's password.
- Superadmin protections already exist via `is_superadmin_user()` and are enforced for role/status/delete actions.
- Password change logic exists for the logged-in admin only (`admin/change-password.php` and helper in `includes/admin.php`).

## Proposed Scope
### In scope
1. Add per-user credential edit action in `admin/users.php`.
2. Add per-user password reset/change action in `admin/users.php`.
3. Enforce protection rules so superadmin accounts cannot be changed.
4. Log all new admin actions to `System_Logs`.
5. Add UI controls in the Users table action panel.

### Out of scope (unless explicitly requested)
- Bulk credential edits
- Sending email notifications for credential/password changes
- Forcing password reset on next login (no such flag currently present)

## Business Rules and Guardrails
1. **Protected account rule:** If target user is superadmin (`is_superadmin = 1`), block credential/password updates.
2. **Email uniqueness rule:** New email must be valid and not used by another user.
3. **Password policy rule:** New password must be at least 8 characters (align existing app rules).
4. **Self-edit consistency:** Admin can still edit own account through users page (except if account is marked superadmin).
5. **Auditability:** Every success/failure attempt should be logged with clear action types.

## Implementation Plan

### Phase 1: Backend action handlers in `admin/users.php`
Add two new `POST` actions inside existing request handling flow:

1. `update_user_credentials`
   - Inputs: `user_id`, `full_name` (optional based on requirement), `email`
   - Validation:
     - `user_id` > 0
     - `email` format valid
     - if `full_name` included: non-empty after trim
   - Transaction flow:
     - `SELECT ... FOR UPDATE` target user
     - block if not found
     - block if superadmin
     - duplicate email check excluding target user (`WHERE email = ? AND id <> ?`)
     - `UPDATE Users SET full_name = ?, email = ?, updated_at = NOW() WHERE id = ? LIMIT 1`
   - Set flash messages and redirect via PRG pattern.
   - Log action: `user_credentials_update` with `success` or failure code.

2. `admin_reset_user_password`
   - Inputs: `user_id`, `new_password`, `confirm_password`
   - Validation:
     - required fields
     - minimum length >= 8
     - confirm match
   - Transaction flow:
     - `SELECT ... FOR UPDATE` target user (include `is_superadmin`, `password_hash`)
     - block if not found
     - block if superadmin
     - optionally block if new password equals existing hash (`password_verify`)
     - hash via `password_hash(..., PASSWORD_BCRYPT)`
     - `UPDATE Users SET password_hash = ?, updated_at = NOW() WHERE id = ? LIMIT 1`
   - Set flash messages and redirect.
   - Log action: `admin_password_reset` with `success` or failure code.

### Phase 2: UI updates in `admin/users.php`
In each non-superadmin row action panel (`Role & delete` section), add:

1. **Edit credentials form**
   - Prefilled with current `full_name` and `email` (or email-only if narrowed scope).
   - Hidden fields: `csrf_token`, `action=update_user_credentials`, `user_id`.
   - Submit button label: `Save credentials`.

2. **Reset password form**
   - Inputs: `new_password`, `confirm_password`.
   - Hidden fields: `csrf_token`, `action=admin_reset_user_password`, `user_id`.
   - Submit button label: `Update password`.

3. Keep existing superadmin display as non-interactive `Protected account`.

### Phase 3: Client-side confirmation and UX polish
1. Extend existing JS submit handlers to include:
   - confirmation prompt (or SweetAlert) for password reset action
   - duplicate/no-change hints for credential form where possible
2. Maintain current accessibility patterns (`aria-live`, disabled submit on submit).
3. Reuse flash success/error mechanism already present on page.

### Phase 4: Audit logging standardization
Use `log_admin_action()` for new actions with clear outcomes:
- `user_credentials_update`
- `admin_password_reset`
- failure examples:
  - `failure:user_not_found`
  - `failure:protected_superadmin`
  - `failure:duplicate_email`
  - `failure:validation`

## Files Expected to Change
1. `admin/users.php`
   - Add backend handlers
   - Add forms in table row actions
   - Add/extend JS handlers for confirmations and form UX
2. `includes/admin.php` *(optional refactor)*
   - Optionally extract helper(s) to avoid duplicating reset logic
   - Keep optional to reduce scope if direct implementation in `admin/users.php` is preferred

## Test Plan

### Functional tests
1. Admin updates borrower/librarian/admin (non-superadmin) email successfully.
2. Admin tries duplicate email and gets validation error.
3. Admin resets borrower/librarian/admin (non-superadmin) password successfully.
4. Admin tries mismatched password confirmation and gets validation error.
5. Admin attempts credential/password update on superadmin and is blocked.

### Regression tests
1. Existing role change still works.
2. Existing activate/deactivate still works.
3. Existing delete protections still work.
4. User login works with updated credentials/password.

### Security checks
1. CSRF required for all new forms.
2. SQL remains parameterized.
3. No plaintext password logging/storage.
4. Superadmin remains immutable for these operations.

## Acceptance Criteria
1. Admin can update credential fields for all non-superadmin users.
2. Admin can reset password for all non-superadmin users.
3. Any attempt to change superadmin credentials/password is blocked and logged.
4. Flash + audit logs clearly show outcome of each action.
5. No regression in existing user management CRUD actions.

## Recommended Default Interpretation
Treat "user credential" as **email + full name** (username-equivalent identity in this system), since this codebase authenticates by email and displays full name as primary identity.
